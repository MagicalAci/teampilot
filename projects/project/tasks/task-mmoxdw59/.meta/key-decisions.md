## 决策清单

### 1. 输入输出结构

- **背景**: 用户明确输入为「走神专注度图片、科目、时间」，输出为「总结文本」。
- **选择**: 输入字段定为 `focus_image`（URL 或 base64）、`subject`、`time_range`；输出为单字段 `summary`（一段总结文本），接口返回 `{"summary": "..."}`。
- **理由**: 与 brief 一致，便于前端直接展示；单字段减少解析成本。
- **替代方案**: 曾考虑多字段（如 summary + suggestion），为与「希望得到一个总结文本」对齐而采用单字段。

### 2. 缺图与回退口径

- **背景**: 需明确缺图、缺科目/缺时间时的行为。
- **选择**: 缺 `focus_image` 时直接报错「缺少专注度图片」，不调用模型；缺 `subject`/`time_range` 时用占位「未指定科目」「未指定时间」。
- **理由**: 无图则无法生成有意义的专注度总结；科目与时间为辅助信息，可回退。
- **替代方案**: 缺图时返回空总结会误导业务，故采用硬校验。

### 3. Mock 模式与测试跑通

- **背景**: 需在无真实 API 环境下跑通脚本测试并产出可验收的测试输出。
- **选择**: 在 client/summary_generator 中增加 `use_mock` 与 `USE_MOCK=1`，Mock 时返回含科目、时间的固定模板总结；新增 `run_tests.py`（unittest）、多组测试数据与 `output_test_run_*.json`。
- **理由**: 保证 CI/本地均可跑通；测试数据包含 URL、base64、多科目，覆盖主要输入形态。
- **替代方案**: 仅用 pytest mock 不跑真实链路；当前采用 Mock 端到端 + 落盘输出，便于产品与开发验收。

### 4. 测试数据中的「图片」

- **背景**: 用户要求「自己制作测试数据、AI 生产图片等」。
- **选择**: 使用 1×1 透明 PNG 的 base64 作为「图片」占位，写入 `input_base64.json`；Mock 不解析图片内容，仅验证输入形态与输出格式。
- **理由**: 真实识图依赖多模态 API；占位图满足「有图」的输入形态，真实上线时替换为真实专注度图表即可。
- **替代方案**: 可后续接入真实多模态接口后，用真实截图再跑一轮端到端。
